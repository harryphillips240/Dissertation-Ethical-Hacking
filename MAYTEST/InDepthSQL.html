An in depth look - SQLi

How is it done

Injection at its core is the abuse of sending data, while you may have an input form expecting input of a certain type, an attacker can do unexpected things they shouldn’t be able to do. There are multiple types of injection, with SQL injection being the most common and my focus for this vulnerability.

In SQL injection an attacker sends SQL that can read or modify a database, this can have massive effects such as gaining users username and passwords, or even deleting whole tables in the database. Applications are particularly vulnerable to this attack when user input is not validated correctly and not sanitised.

Insecure code example: Here we just have a textbox value going straight into a statement, allowing the user to break the statement with a ' and then run their own statement. Scenarios

There are many scenarios where SQL injection could be used. As long as there is a database on a site with insecure code passing statements to it then its likely you can get whatever information you want from the database, including user information. You could find the admins account information and have full power, you could insert new items or update existing ones, even drop or add completetly new tables to the database.

Imagine a school system with poorly implemented SQL, you could find the table containing your grades and update them to be a different number, or you could drop the whole table so the school loses everyones work. The possibilities are endless.

Ethics

Now that you know what SQLi is, how to check if its possible on sites, and how to do it yourself, its important to talk about the implications of this knowledge.

While SQLi can be fun to experiment with, its important to only do this in places like this vulnerable web app, and to never do it on a live site. In fact, if you find that a site is vulnerable to SQLi you should stay away from the site and notify the admins because none of your data is safe and neither is anyone elses.

The results of malicious users using SQLi can be massive, as mentioned earlier you could change or drop the tables of a database, depending on the context potentially ruining thousands of peoples saved data or you could steal their data for yourself. Using SQLi maliciously will get you into big legal trouble even if you dont mean any harm so be sure to keep it safe and ethical.
